# Code of Conduct

Orbix is an open-source project for people building and running self-hosted media software. We expect project spaces to stay practical, respectful, and safe for contributors and users.

## Expected Behavior

- Be respectful of different backgrounds, experience levels, and technical preferences.
- Keep discussion focused on the project, the user impact, and the evidence behind a proposal.
- Give and receive technical feedback in good faith.
- Assume public project spaces are shared workspaces, not private arguments.
- Respect privacy. Do not post private logs, secrets, personal information, or identifying media paths.

## Unacceptable Behavior

Unacceptable behavior includes:

- Harassment, threats, insults, or sustained personal attacks.
- Sexualized language or attention in project spaces.
- Discriminatory language or behavior.
- Doxing or publishing private information without explicit permission.
- Deliberate intimidation, stalking, or unwanted repeated contact.
- Trolling, spam, or repeated disruption of maintainers' work.
- Using project infrastructure to share illegal, harmful, or abusive material.

## Scope

This Code of Conduct applies in all Orbix project spaces, including GitHub issues, pull requests, discussions, reviews, documentation, chat channels if added later, and maintainer-run community events.

It also applies when someone represents the project in public, such as using an official project account or speaking on behalf of maintainers.

## Reporting

Report Code of Conduct concerns privately to:

```text
conduct@orbix.example
```

This is a placeholder contact for now. Replace it with the real project email or private reporting channel before a public community launch.

Include as much context as you safely can: links, screenshots, dates, usernames, and a description of what happened. Do not report security vulnerabilities through this address; use [SECURITY.md](SECURITY.md) instead.

## Enforcement

Maintainers will review reports privately and proportionately. Depending on severity and context, maintainers may:

- Ask someone to edit or remove content.
- Warn a participant.
- Temporarily limit participation.
- Close, lock, or hide disruptive threads.
- Ban a participant from project spaces.
- Escalate to the platform or another appropriate authority when needed.

Maintainers who are directly involved in a report should recuse themselves from handling it when another maintainer is available.
